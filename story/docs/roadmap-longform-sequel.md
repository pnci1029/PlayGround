# 로드맵 — 장편(long-form) · 속편(sequel) 가능성 진단

> 현재 MVP는 **단편**(4챕터, ~2,000–3,000자, OpenAI 2콜: 아웃라인→산문). 두 확장 기능의 구현 가능성·방법·비용·작업량을 정리. (기법 근거: `llm-fiction-research.md`)

---

## 1. 장편 (Long-form)

### 결론: **가능. 단 MVP보다 훨씬 큰 작업** → Phase 2, 단계적으로.

### 왜 단순 확장이 안 되나
- **max_tokens 한계**: 한 번의 응답에 장편 전체를 못 담음 → **챕터별 개별 호출** 필수.
- **연속성 붕괴**: 호출이 많아질수록 인물·설정·플롯이 어긋남(리서치 §2.3 "Lost in Stories" — 중반부·고엔트로피 구간에서 오류 집중). → **메모리/상태 추적** 필요.
- **비용·지연**: 장편 1편 = 단편 여러 편치. 하루 3개 쿼터와 충돌(장편 1개를 단편 3개처럼 칠지 정책 필요).

### 접근 (검증된 방법, 리서치 §2.2)
1. **계층적 생성**: 전체 거친 아웃라인 → 챕터별 상세 아웃라인 → 챕터별 산문 (DOME/DOC 방식).
2. **Story Bible(상태객체)**: 인물·설정·타임라인을 JSON으로 유지, 매 챕터 호출 앞에 주입 + 생성 후 갱신 (RecurrentGPT 루프).
3. **직전 요약 주입**: 장기기억(누적 요약) + 단기기억(직전 챕터 요약).
4. **비동기 생성**: 오래 걸리므로 작업 큐 + 진행률/스트리밍 UX (생성 중 이탈해도 이어서).

### 스키마 영향
- `stories`에 `length_type VARCHAR(10)`('short'|'medium'|'long'), `bible JSONB`(상태객체) 추가.
- 챕터가 많아지면 `chapters` JSONB 대신 별도 `story_chapters(story_id, idx, title, body)` 테이블 고려.

### 작업량 / 추천
- **큼**(메모리·연속성·비용관리·비동기). 한 번에 "장편"은 위험.
- **권장 단계**: 먼저 **중편**(8–10챕터) — 단일 전체 아웃라인 → 챕터별 호출 + "직전 챕터 요약"만 주입(가벼운 메모리). 이게 안정되면 story bible·비동기로 진짜 장편 확장.

## 2. 속편 (Sequel)

### 결론: **비교적 쉬움. 단편 파이프라인 재사용** → 우선순위 높게 곧 구현 가능.

### 개념
기존 이야기 X를 읽고 "속편 쓰기" → X의 로그라인·결말·인물을 컨텍스트로 주입해 **이어지는 새 이야기 Y** 생성(같은 장르 유지/선택).

### 구현 방법 (기존 구조에 작게 얹기)
- **스키마**: `stories`에 `parent_id UUID`(속편이면 원작 id, 아니면 NULL) 추가. (`ALTER TABLE ... ADD COLUMN IF NOT EXISTS` 멱등 — premise 컬럼 추가 때와 동일 패턴)
- **API**: `POST /api/stories/:id/sequel` (body `{ premise? }`, 선택적 방향 입력)
  - 원작 로드 → (원작이 길면) 요약 1콜 → outline 프롬프트에 "원작 줄거리/결말 요약 + 인물"을 주입 → 기존 `generateOutline`/`generateProse` 그대로 → 새 story 저장(`parent_id` 설정, 쿼터 +1).
- **프롬프트**: 시스템에 "다음은 원작의 줄거리와 결말이다. 이를 자연스럽게 잇는 속편을 써라. 원작 인물·설정과 모순되지 않게." + 원작 요약 블록.
- **UX**: 리더에 **"속편 쓰기"** 버튼 → 생성(스피너) → 새 리더로. (사용자가 방향을 한두 문장 적게 할 수도 — `/create?parent=<id>` 재사용)
- **쿼터**: 속편도 1개 카운트(기존 정책 그대로).

### 작업량 / 추천
- **작음~중간**. 단편 파이프라인 + 원작 컨텍스트 주입 + `parent_id` 1컬럼 + 라우트 1개 + 버튼 1개.
- **권장**: 장편보다 먼저. 단편 기반이라 리스크 낮고 사용자 가치 큼(시리즈 느낌).

---

## 우선순위 제안
1. **속편** (작고 가치 큼, 곧 구현 가능)
2. **중편**(8–10챕터, 가벼운 메모리) — 장편의 디딤돌
3. **진짜 장편**(story bible + 비동기 생성) — Phase 2 본격 과제

> 비용 메모: 장편/중편은 호출 수가 늘어 **하루 쿼터 정책**(장편을 단편 몇 개로 칠지)과 **OpenAI 예산 캡**을 같이 정해야 함.
