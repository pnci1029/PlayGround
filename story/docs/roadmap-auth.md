# 로드맵 — 로그인 / 계정 (필요성 & 설계 진단)

> 현재 신원 = **익명 디바이스 UUID**(localStorage `story_uid`, 헤더 `X-Story-Uid`). 기기/브라우저 단위라 한계가 명확. 정식 출시 전 계정 도입 검토. (작성: 베타 기간)

## 왜 필요한가
- **기기/브라우저 바뀌면 내 글·쿼터·공개권한 상실** (캐시 삭제·시크릿모드도 동일). 네트워크(와이파이) 변경은 무관.
- 크로스 기기 연속성, **진짜 비공개(소유권)**, 쿼터 우회 어뷰즈 방지를 위해 계정이 있어야 신뢰도↑.

## 접근안

### A. 소셜 로그인 + 익명 uid 머지 (권장, 정식용)
- **카카오/구글 OAuth.** 로그인 시 계정 생성, 현재 localStorage `story_uid`의 기존 글들을 그 계정으로 **귀속(merge)**.
- 스키마: `story.users(id, provider, provider_id, nickname, created_at)`; `stories`에 `user_id UUID NULL` 추가(익명 글은 NULL 유지). 머지 = `UPDATE story.stories SET user_id=? WHERE author_uid=? AND user_id IS NULL`.
- 쿼터/권한: 로그인 사용자는 `user_id` 기준, 비로그인은 기존 `author_uid` 기준(병행).
- 참고: 모노레포 `blog`가 JWT 로그인 패턴 보유 → 토큰 발급/미들웨어 재사용 가능. story는 소셜이 더 친화적.
- 작업량: **중간**(OAuth 콜백, users 테이블, 머지 로직, FE 로그인 UI, 토큰 미들웨어).

### B. 복구 코드 (초간단, 로그인 없이 — 당장 임시방편)
- 가입 없이 현재 `uid`를 "복구 코드"로 보여주고, 다른 기기에서 그 코드를 입력 → localStorage에 세팅해 **동일 신원 복원**.
- 장점: 구현 매우 쉬움. 단점: 코드 분실 시 복구 불가, 보안 약함(코드 알면 사칭 가능).
- 작업량: **작음**.

### C. 매직링크(이메일) — 중간
- 이메일로 1회용 로그인 링크. 비번 없는 계정. OAuth보다 셋업 단순, 이메일 발송 인프라 필요.

## 권장
- **정식 출시 전: A (소셜 로그인 + uid 머지).** 쿼터·소유권·연속성 모두 해결.
- **그 전 임시로는: B (복구 코드)** 도 가능.
- 진짜 비공개(작성자 외 열람 차단)도 계정 신뢰성이 있어야 의미 있음 → A와 함께 강화.

## 작가 표시 = 닉네임 (요구사항)
- 가입(로그인) 시 **닉네임**을 부여하고, 글의 **작가를 닉네임으로 표시**한다(피드·리더·내 글에서 "by 닉네임").
- **닉네임은 수정 가능**해야 한다(설정에서 변경 → 기존 글에도 반영, `user_id` 기준 조인이라 자동 반영).
- 스키마: `story.users.nickname VARCHAR` + 유니크/중복 정책(중복 허용 or 유니크는 결정 필요). 표시는 `stories JOIN users ON user_id`.
- API: `PATCH /api/me`(닉네임 변경), 응답·목록에 `author_nickname` 포함.
- 비로그인(익명) 글은 작가 표시 없음(또는 "익명").

## 영향 (구현 시)
- **DB**: `story.users` 신규, `stories.user_id` 추가, 익명 uid 머지 마이그레이션.
- **API**: `/api/auth/*`(OAuth 콜백·토큰), 기존 `X-Story-Uid`와 `Authorization` 병행 → 점진 전환.
- **쿼터**: `daily_usage` 키를 `author_uid` → 로그인 시 `user_id`로 전환.
- **FE**: 로그인 버튼/상태, 토큰 저장(localStorage), 헤더 첨부.

## 상태
- **미구현.** 본 문서는 설계·가능성 정리. 베타 동안은 익명 uid 유지 + 작성 수 무제한(`STORY_UNLIMITED`).
